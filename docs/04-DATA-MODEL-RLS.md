# 04 - Data Model & Security (RLS)

Roommate Link uses a strictly relational model in PostgreSQL. Security is enforced at the database layer via **Row Level Security (RLS)**, ensuring that even if a malicious user bypasses the React UI, they cannot access other students' private data.

## 🆔 The Split-ID Architecture
One of the most frequent sources of bugs is the two-tier ID system:

1.  **`auth.users.id` (Auth ID):** The UUID generated by Supabase Auth. This is used for authentication and session management.
2.  **`public.users.id` (Profile ID):** The primary key of our custom users table.
3.  **The Link:** Most tables in the `public` schema (like `matches` and `messages`) reference the **Profile ID** (`public.users.id`). 

**The Rule:** When querying, you must often fetch the profile first using the Auth ID:
```sql
-- CORRECT
SELECT * FROM messages WHERE sender_id = (SELECT id FROM users WHERE auth_id = auth.uid());
```

## 📊 Core Tables

### `public.users`
The heart of the application. Tracks payment status, verification, and bio.
- `has_paid`: Boolean. Controls access to match details.
- `is_pioneer`: Boolean. Identifies early-adopter students.
- `status`: Enum (`ACTIVE`, `COMPLETED`, `EXPIRED`, `HIDDEN`, `SUSPENDED`).

### `public.matches` (The A/B Ambiguity)
The algorithm calculates matches in two directions, but only one record exists per pair.
- **Ambiguity:** A user can be either `user_a_id` OR `user_b_id` in the `matches` table.
- **Frontend Fix:** Always query BOTH columns and deduplicate by the roommate's ID.

### `public.questionnaire_responses`
Stores the raw vector of answers (JSONB) used by the matching engine.

## 🛡️ Security Protocol: RLS
We follow a **"Locked by Default"** policy.

### Message Security
- **SELECT:** `auth.uid() = sender_id OR auth.uid() = receiver_id`.
- **UPDATE:** Limited to `receiver_id` only (marking messages as `READ`).

### Profile Security
- **SELECT:** Public profiles are visible to all authenticated users, but sensitive fields (like `student_email`) are protected or only visible to the owner.
- **INSERT/UPDATE:** `auth_id = auth.uid()`. Cross-user editing is physically impossible at the database level.

## 📜 Migrations Protocol
We use a **Version-Controlled Migration** system.
- **Path:** `supabase/migrations/`
- **Rule 1:** Never use `DROP TABLE` in production.
- **Rule 2:** Use `ALTER TABLE` with `IF NOT EXISTS` for all additions.
- **Rule 3:** RLS policies must be scripted in SQL; they should not be managed manually via the dashboard.
